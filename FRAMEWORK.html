What You Get

A single source of truth that Claude Code reads before every action. Your entire company — revenue, customers, pipeline, product, support — accessible in plain language. Ask questions. Give orders. Everything gets executed.

Ask anything
  • How's my revenue this month?
  • What's the main reason customers are churning?
  • What's the most requested feature we don't have?
  • What's the most common support problem?
  • Which accounts are at risk right now?
  • What are my biggest open deals this month?
Give orders
  • Follow up with users who haven't made a 2nd query after 7 days
  • Send a thank-you email to customers with 100+ queries and 6+ months with us
  • Email accounts above $500/mo introducing a dedicated account manager
  • Generate a weekly report of at-risk accounts with suggested next actions
  • Draft a win-back sequence for customers who churned in the last 30 days

Getting Started

// for you — human

Place this file in your project root. Open Claude Code and say:

"Help me implement the Claude Code OS for my company."

The agent reads this file, understands the structure and guides you through the entire setup in plain language.

Setup Protocol — For the Agent

If you are an agent reading this after the activation command: execute the flow below in order. One step at a time.

// bootstrap_protocol · v1.1
01
Check the context

Verify whether _brain/index.html already exists. If yes: ask restart or expand? If no: fresh setup — continue. Also check whether a _brain/inbox/ folder exists with files already dropped by the user — if yes, jump to step 02 before asking any questions.

02
Knowledge Drop — process everything you already have

Create _brain/inbox/ and tell the user: "Drop anything here — documents, files, code, configs, entire project folders. SOPs, brand guide, price tables, contracts, codebase, .env files, anything. I'll read everything, extract what matters, and build the brain from what you already have. Minimum effort required."

Once files are present, read each one and extract knowledge. Map content to the correct brain files automatically (brand guide → core/brand, price table → core/product, SOP → skills/, code → development/). For large codebases, process in stages: architecture and entry points first, critical modules next, depth on demand — never try to read everything at once. Check for .env files and existing API configurations — use those connections directly without asking the user to reconnect tools that are already configured. After processing, move all files to _brain/archive/YYYY-MM-DD/.

If no files are dropped after 60 seconds, continue to step 03 — the interview will fill the gaps.

03
Smart interview — fill gaps and clarify ambiguities

Based on what was found (or not found) in the inbox, ask only the questions that are still unanswered. One at a time. Never a form. Never all at once.

  1. Identity What is the company name and what does it do in one sentence? (if not found in documents)
  2. Product What is the main product or service and how does it work for the customer?
  3. Customers Who are the customers? Profile, segment, average size.
  4. Teams Which teams or areas exist? (Dev, Sales, Marketing, Support, Ops, Finance, Legal…)
  5. Tools Which tools are you using today? List what you have — payments, database, CRM, analytics, email, support…
  6. Auto-update Which of those tools can connect directly? (e.g. Stripe for revenue, Supabase for usage) — I'll set up the crons automatically.

After each tool answer, proactively suggest better integrations if detected: "You mentioned exporting CSVs from your database — I can connect directly to Supabase and update the metrics automatically. Want me to set that up?"

04
Proactive gap questions

Based on everything collected so far — documents + interview — identify what's still missing or ambiguous and ask targeted follow-up questions before building anything. Examples:

  1. Clarify "Your brand guide mentions a primary color but no hex code — what's the exact value?"
  2. Clarify "Your price table has 3 plans but no rate limits per plan — what are they?"
  3. Clarify "Your SOP mentions a CRM tool but doesn't name it — which one?"
  4. Suggest "I noticed you don't have a churn tracking process — want me to create a skill for that?"
  5. Suggest "You mentioned daily CSV exports — connecting directly to [tool] would be faster. Should I set it up?"
05
Adapt the structure and show the plan

Based on all collected information, decide which folders and files to create. Remove what doesn't apply (no sales team = no sales/). Add what's needed (legal/, hr/, finance/). Show the plan to the user in a clear tree and wait for approval before creating anything.

06
Build the brain + skills

Create all brain files with real content from the collected knowledge — no blank placeholders. Use the timestamp header pattern from this document. Also create initial skill files in _brain/skills/ for any recurring processes identified during the interview or in the inbox documents.

07
Update CLAUDE.md and set up crons

Add the full brain + skills protocol to the root CLAUDE.md (create it if missing). Then set up all approved cron jobs — one command per integration. Confirm each cron is running before moving on.

08
Propose dashboard

Based on the metrics and tools connected, propose a _brain/dashboard.html that shows the company's north star metrics at a glance. Auto-refreshes in the browser. Updated by the same cron jobs. Ask: "Want me to build the dashboard now?"

09
Validate and deliver

Show a summary of everything created. Ask two test questions — one about product or pricing, one about a customer or metric. The agent must answer from the brain files, never from memory. If it guesses, the brain is incomplete — fix before closing setup.

Philosophy

PrincipleWhat it means in practice
index-firstThe agent always reads index.html before any action. The index maps where everything lives.
on-demand loadingOnly the relevant sub-file is read per task. Reduces token usage and eliminates hallucination from unnecessary context.
living brainOperations files are automatically updated by cron jobs connected to Stripe, Supabase, and other sources.
human visibilityEvery file has a header with timestamp ("3h ago") and a countdown to the next update. Stale data is immediately visible.
embedded promptEach file carries its own update prompt. Anyone — human or agent — knows what to do without reading external docs.
skills layerRecurring instructions are saved to _brain/skills/ as operational playbooks. The agent updates them automatically when the user corrects or refines its behavior.
memory ≠ skillsMemory stores facts about the company and user. Skills store how to operate — rules, tone, filters, playbooks. Different purpose, different files.
knowledge dropRaw documents (SOPs, brand kits, price tables, contracts) go into _brain/inbox/ — during setup or anytime. The agent checks the inbox at the start of every session, processes any new files, updates the relevant brain files, and archives the originals to _brain/archive/YYYY-MM-DD/.
proactive gapsAfter reading inbox files, the agent identifies what's missing or ambiguous and asks targeted follow-up questions — not a generic form, but specific clarifications based on what it actually found.
one fileThe entire framework is a single self-contained HTML file. Drop it in any project, say the trigger phrase, and the agent builds the full brain from scratch.

AI Protocol — CLAUDE.md

Add this block to the root CLAUDE.md of your project. This ensures the agent reads the index before any action.

## The Claude Code OS — Mandatory Protocol

Before any task, follow this protocol:

1. Read `_brain/index.html` — it is the map of all company knowledge
2. Identify which sub-files are relevant to the current task
3. Read ONLY those files — do not load unnecessary context
4. For operational tasks (outreach, content, analysis), also load the relevant `_brain/skills/` file
5. If you generate new relevant information, update the corresponding file
6. Log every change in `_brain/logs/updates.html`

Skills auto-update rule:
When the user corrects your approach or gives a recurring instruction ("always do X", "never do Y",
"from now on..."), immediately update the relevant `_brain/skills/[area].md` file.
If no skill file exists for that area, create one.
Add a timestamp and note what changed and why.

Never: guess information that exists in the brain files.
Always: consult the brain before answering about product, pricing, customers or roadmap.
Always: consult `_brain/skills/` before any recurring operational task.
Rule: information not in the brain is unreliable — ask the user to update it.

<script>
// ⚠️  Only change these two lines when saving an update
const LAST_UPDATED = '2026-05-29T20:00:00-03:00';  // ISO 8601
const NEXT_UPDATE  = '2026-05-30T08:00:00-03:00';  // next cron or manual date

function timeAgo(d) {
  const s = Math.floor((Date.now() - d) / 1000);
  if (s < 60)    return s + 's ago';
  if (s < 3600)  return Math.floor(s / 60) + 'min ago';
  if (s < 86400) return Math.floor(s / 3600) + 'h ago';
  return Math.floor(s / 86400) + 'd ago';
}
function countdown(d) {
  const diff = d - Date.now();
  if (diff <= 0) return 'pending';
  const h = Math.floor(diff / 3600000);
  const m = Math.floor((diff % 3600000) / 60000);
  return 'in ' + (h ? h + 'h ' : '') + m + 'min';
}
function fmt(d) {
  return d.toLocaleString('en-GB', { day:'2-digit', month:'2-digit', hour:'2-digit', minute:'2-digit' });
}
function render() {
  const last = new Date(LAST_UPDATED), next = new Date(NEXT_UPDATE);
  document.getElementById('last-abs').textContent = fmt(last);
  document.getElementById('last-rel').textContent = timeAgo(last);
  document.getElementById('next-abs').textContent = fmt(next);
  document.getElementById('next-rel').textContent = countdown(next);
  document.getElementById('stamp-last').className =
    'stamp ' + (Date.now() - last < 86400000 ? 'fresh' : 'stale');
}
render();
setInterval(render, 30000); // refresh every 30s
</script>

Skills Layer — Operational Playbooks

Skills are the layer between knowledge (brain) and action (Claude Code). They store how to operate — tone, rules, filters, sequences — for every recurring task. Unlike the brain files (which store facts), skills store instructions that evolve as the user corrects or refines the agent's behavior.

// auto-update trigger Any time the user says "always do X", "from now on...", "never do Y", or corrects a recurring pattern — the agent immediately writes or updates the relevant skill file. No manual step required.

How to Replicate for Another Company

  1. Drop FRAMEWORK.html in the project root. That's the only file needed to start — no folder structure, no CLAUDE.md, nothing else.
  2. Say the trigger phrase — "Help me implement the Claude Code OS for my company." The agent reads this file and runs the Bootstrap Protocol automatically.
  3. Drop raw documents into _brain/inbox/ when prompted — SOPs, brand guide, price tables, FAQs, contracts, anything. The agent reads and organizes everything.
  4. Answer the smart interview — one question at a time, in plain language. The agent only asks what it couldn't find in the documents. It also suggests better tool integrations proactively.
  5. Approve the structure plan and cron connections. The agent proposes, you approve. One word is enough for each decision.
  6. Done. Brain is live, crons are running, dashboard is built. From now on, talk naturally — give orders, ask questions, correct approaches. The agent manages everything.
